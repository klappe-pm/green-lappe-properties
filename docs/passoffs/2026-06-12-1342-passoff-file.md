## Blocker findings (the core deliverable of this session)

1. **No website code existed.** This repo was a docs/decision-support corpus.
   The design system says site code lives in separate `green-pm-site/`,
   `green-pm-studio/`, `green-pm-operations/` repos that do not exist. User
   decision: build it here, organized around features, until it scales.
2. **Brand/domain conflict (now resolved).** The 2026-05-20 website-domain
   launch plan recommended name *Green Lappe* / `greenlappe.com`; the locked
   design system v3 (2026-05-21) uses *Green Property Management / Green PM* /
   `greenpmpnw.com`. User decision: **Green PM / greenpmpnw.com is canonical**;
   treat the `greenlappe.com` note as superseded.
3. **Stack: locked vs. open (now reconciled by decision).** The launch plan
   treats stack/hosting as open; the design system already locked
   Astro + Tailwind + Sanity + Cloudflare Pages + HubSpot + Postmark + Clerk +
   Neon (`docs/uxd/design-system/docs/80-system-architecture.md`). Treat the
   design system as canonical; the launch plan's "open decisions" table is
   stale.
4. **Missing `.claude/` IA tooling.** `CLAUDE.md` and `.githooks/pre-commit`
   both require `.claude/evaluate-ia-change.mjs` and
   `.claude/information-architecture.md`, but `.claude/` is gitignored/untracked
   and absent from fresh clones (it was intentionally untracked per the
   2026-06-07 passoff). The mandated validation command cannot run in a fresh
   clone / web session. `core.hooksPath` is not set, so the pre-commit hook does
   not fire automatically — but local checkouts that wire it up will fail.
5. **External gates still genuinely blocked:** counsel-approved public language,
   licensing/firm status, domain purchase, and public-launch approval remain
   owner/counsel/account-owner actions per the website launch plan.

# Key decisions

- **Canonical brand/domain:** Green Property Management / "Green PM" /
  `greenpmpnw.com` (user-selected; design system v3 wins over the older launch
  note).
- **Repo strategy:** the website is built in THIS repo under `site/`, organized
  around features, until scale justifies splitting into the design system's
  separate repos. (User: "all of it. this is the project.")
- **Design tokens are copied, not re-authored.** `docs/uxd/design-system/`
  remains the single source of truth; `site/src/styles/` holds copies, and
  Tailwind preflight is disabled so `base.css` is the only reset.
- **Did not re-track `.claude/`.** The prior passoff's "Do not do" forbids
  staging the ignored local IA machinery without explicit user request, so the
  IA-tooling blocker is documented rather than fixed by re-adding files.
- **Did not invent any blocked decision** (legal, licensing, vendor, domain
  purchase, screening provider, etc.), per `CLAUDE.md` boundaries.

# Risks and open questions

- The `site/` scaffold is **not yet buildable**: there is no `src/pages/` route
  or base layout, so `astro build` will fail until a homepage + `BaseLayout` are
  added. Do not treat `site/` as a working app yet.
- Dependencies are declared but **not installed and the build was not run/
  verified** this session. Versions in `package.json` are reasonable but
  unverified against an actual install.
- The IA validation command in `CLAUDE.md` could not be run (no `.claude/`).
  Changes in this session were not validated by the deterministic evaluator.
- The design-system inventory surfaced ~20 TBD/contradiction items (screening
  provider, Rentvine-vs-Postmark email sender, `page.sections` field specs,
  renter-product surface tokens, blog pagination as query-string vs path, etc.)
  that must be resolved before feature specs are final. Full list is in the
  session transcript; reproduce by reading `docs/uxd/design-system/docs/`
  files 81/82/83/84/85/50/51/11.
- The two older website plans
  (`docs/launch/plans/2026-05-20-website-domain-internal-launch-plan.md`,
  `2026-05-21-website-phase-completion-register.md`) still reference
  `greenlappe.com` and an open stack; they need a "superseded by" banner pointed
  at the new canonical plan.

# Lessons learned

- The single biggest blocker was not technical but a governance conflict: two
  canonical docs disagreed on brand name and domain. Resolving that with the
  user first prevents writing a large spec/test suite against the wrong brand.
- `npm` registry is reachable from this environment (`npm ping` succeeded,
  Node v22.22.2), so an install + build verification IS possible in a future
  session — it just wasn't reached before `/pass`.
